## 测试策略总览

### 测试环境
- **测试方式**：静态代码审查 + 接口规范分析（因当前环境为代码审查阶段，无法直连数据库/启动服务）
- **覆盖维度**：正向流程、异常入参、边界值、权限校验、逻辑完备性、安全合规

### 缺陷等级定义
| 等级 | 定义 |
|------|------|
| **P0-阻塞** | 核心业务流程不通，导致无法使用 |
| **P1-严重** | 业务逻辑错误、数据异常、权限绕过 |
| **P2-一般** | 边界未处理、交互体验问题、缺少校验 |
| **P3-轻微** | 规范问题、代码风格、日志不足 |

### ⚠ TC-PAY-001：Mock支付模式安全风险
| 项目 | 内容 |
|------|------|
| **用例ID** | TC-PAY-001 |
| **接口** | POST /api/payments/mock/{paymentNo}/success |
| **测试类型** | 安全 |
| **风险分析** | Mock支付接口可能无身份验证，任何人知道 paymentNo 即可模拟支付成功。这会绕过真实支付流程。 |
| **缺陷等级** | **P0-阻塞** |
| **优化建议** | Mock支付接口必须：① 仅在开发/测试环境可用(profile校验)；② 生产环境要彻底禁用或添加认证 + IP白名单；③ paymentNo 使用不可猜测的 UUID 而非自增ID |

### ⚠ TC-PAY-002：支付结果通知幂等性缺失
| 项目 | 内容 |
|------|------|
| **用例ID** | TC-PAY-002 |
| **测试类型** | 并发 |
| **风险分析** | 支付回调处理未实现幂等性保障。若网络抖动导致重复回调，可能：① 重复扣减余额；② 重复创建订阅；③ 重复记录 transaction。 |
| **缺陷等级** | **P1-严重** |
| **优化建议** | 实现回调幂等性：① 以 paymentNo + eventId 作为去重键；② 使用 Redis 分布式锁确保同一 paymentNo 不会被并发处理 |

| **风险分析** | setting_value 类型为 text，无长度限制。若传入超大数据(如 10MB 字符串)，可能导致：① 数据库膨胀；② 查询性能下降；③ 前端显示异常 |
| **缺陷等级** | **P2-一般** |
| **优化建议** | 根据实际业务场景限制每个 key 对应的 value 最大长度(如 65535 字符)或在前端限制输入框长度 |

## 安全 & 架构综合审查

### ⚠ SEC-001：双服务数据库重复定义
| 项目 | 内容 |
|------|------|
| **用例ID** | SEC-001 |
| **类型** | 架构问题 |
| **风险分析** | order_main 表同时存在于 api 模块(V004 migration)和 order-service 模块(V001 migration)。payment_transaction 表也同时存在于两个模块。这是微服务拆分过程中遗留的重复定义问题。未来如果两个模块独立部署，各自操作自己的数据库表，数据将出现分裂。 |
| **缺陷等级** | **P1-严重** |
| **优化建议** | 确定表的归属权：每个表只应属于一个微服务。建议将所有订单和支付表迁移到 order-service 和 payment-service 独享，api 模块通过 Feign 调用而非直接操作数据库。或作为过渡方案，在 api 模块中移除重复的表定义。 |

### ⚠ SEC-002：统一异常处理缺失
| 项目 | 内容 |
|------|------|
| **用例ID** | SEC-002 |
| **类型** | 异常处理 |
| **风险分析** | 各微服务是否实现了统一的全局异常处理(@RestControllerAdvice)？DB层抛出的原始异常(如 ConstraintViolationException、DataIntegrityViolationException) 未捕获时可能返回 500 + 堆栈信息暴露敏感细节。 |
| **缺陷等级** | **P2-一般** |
| **优化建议** | 每个微服务实现全局异常处理器，统一返回格式 `{code, message, data}`，生产环境隐藏堆栈细节 |

### ⚠ SEC-003：CORS 与 CSRF
| 项目 | 内容 |
|------|------|
| **用例ID** | SEC-003 |
| **类型** | 安全 |
| **风险分析** | 前端(3000/5173)与后端(8080)跨域。需确认 Gateway 是否已配置 CORS 白名单而非 `Access-Control-Allow-Origin: *`。此外 Sa-Token 默认启用 CSRF 防护，前后端分离模式下需确认适配。 |
| **缺陷等级** | **P2-一般** |
| **优化建议** | Gateway 配置 CORS 时指定明确的前端域名白名单；前后端分离模式 Sa-Token 使用 token 模式而非 cookie-session 模式 |

### ⚠ SEC-004：SQL注入防护
| 项目 | 内容 |
|------|------|
| **用例ID** | SEC-004 |
| **类型** | 安全 |
| **审查结论** | 项目使用 Spring Data JPA 的 Repository 方法查询，JPA 默认使用参数化查询，基本免疫 SQL 注入。但若存在 @Query 原生 SQL 或自定义 statement 需额外检查。 |
| **实际结果** | ✅ 使用 JPA 方法名派生查询为主，风险较低 |
| **缺陷等级** | **P3-轻微** |

### ⚠ SEC-005：审计日志缺失
| 项目 | 内容 |
|------|------|
| **用例ID** | SEC-005 |
| **类型** | 审计合规 |
| **风险分析** | 当前 audit_log 只记录了 SaaS 用户的操作日志。管理员对敏感操作（CMS发布/套餐修改/用户权限变更）是否记录了操作人、操作时间、操作内容？特别是 product、cms、auth 模块的管理端操作未发现审计日志记录。 |
| **缺陷等级** | **P2-一般** |
| **优化建议** | 所有管理端的 增/删/改 操作应记录审计日志，至少包括：操作人、操作时间、资源类型、资源ID、操作内容快照 |

## 前端审查

### ⚠ FE-001：前端未区分 SaaS Token / Admin Token
| 项目 | 内容 |
|------|------|
| **用例ID** | FE-001 |
| **位置** | frontend/web/ 和 frontend/admin/ |
| **风险分析** | 对外网站和管理后台是两套独立的用户体系（saas_user / admin_user），使用各自的登录接口。但前端如何区分和存储这两套 Token？若 Admin Token 被用于调用公开 API，或 SaaS Token 被用于管理后台，可能产生权限逃逸。 |
| **缺陷等级** | **P1-严重** |
| **优化建议** | ① 明确 Token 的存储 Key 命名分离（admin_token / saas_token）；② API 请求层根据请求前缀(/admin/ vs /api/)自动选择 Token；③ Gateway 层验证 Token 时区分 Token 的 type 属性和目标路由 |

### ⚠ FE-002：前端 API 请求直接调用 order-service/user-service，绕过 BFF
| 项目 | 内容 |
|------|------|
| **用例ID** | FE-002 |
| **位置** | frontend/web/ 调用 /api/orders/**, /api/subscriptions/**, /api/usage/** |
| **风险分析** | 前端直接调用了 order-service 和 user-service 的接口(/api/orders/**, /api/balance/**)而未通过 BFF(api 模块)聚合。这违反了 BFF 模式的设计初衷，可能导致：① 前端需知道多个服务地址；② BFF 层的聚合/转换逻辑被绕过；③ 未来拆分时需修改前端代码。 |
| **缺陷等级** | **P2-一般** |
| **优化建议** | 前端统一通过 BFF 访问，BFF 内部调用微服务获取数据并组装。例如：前端只调 /api/orders 实际走 gateway → api(BFF)，api 内部通过 Feign 调用 order-service |

### ⚠ FE-003：按钮级权限判断使用 v-if 而非 disabled
| 项目 | 内容 |
|------|------|
| **用例ID** | FE-003 |
| **类型** | 用户体验/安全 |
| **风险分析** | 如果无权限按钮使用 v-if 直接移除，用户可能找不到操作入口。更推荐的模式是 disabled + tooltip 提示"无操作权限，请联系管理员"。 |
| **缺陷等级** | **P3-轻微** |
| **优化建议** | 无权限时按钮设为 disabled 并显示 tooltip 提示，而非直接隐藏 |
