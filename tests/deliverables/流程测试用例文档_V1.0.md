## 流程 P001：SaaS用户全生命周期

### TC-P001-N001：正常注册→登录→浏览→购买的完整闭环

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P001-N001 |
| **流程名称** | 用户注册到API调用的全生命周期正常流程 |
| **前置条件** | 方案组+方案+价格+权益已存在且启用；Mock支付模式可用 |
| **操作步骤** | ① 新用户注册（email+password+nickname）→ ② 登录获取Token → ③ 访问定价页查看套餐列表 → ④ 选择方案下单 → ⑤ 发起支付（channel=mock）→ ⑥ 确认支付成功 → ⑦ 查询订阅状态 → ⑧ 获取API Key → ⑨ 调用API记录用量 |
| **预期结果** | ①~⑨ 各步返回正确状态码；订阅自动激活；API Key 可用于鉴权；用量记录写入usage_record |
| **数据校验** | saas_user新增记录；order_main状态=PENDING→PAID→FULFILLED；user_subscription新增active；usage_record新增记录 |

## 流程 P002：下单-支付-订阅闭环

### TC-P002-N001：Mock支付完整闭环

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P002-N001 |
| **流程名称** | Mock创建订单→支付→履约→订阅正常流程 |
| **前置条件** | 用户已登录；产品套餐存在且启用 |
| **操作步骤** | ① POST /api/orders 创建订单 → ② 记录orderNo → ③ POST /api/payments/checkout {orderNo, channel:"mock"} → ④ 得到mockPayUrl → ⑤ POST mockPayUrl 模拟支付成功 → ⑥ GET /api/orders/{orderNo} 查询状态 → ⑦ GET /api/subscriptions/active 检查订阅 |
| **预期结果** | 步骤⑥ status=PAID（最终FULFILLED）；步骤⑦ 订阅 active；order_main.snapshot_json 保存下单时的套餐快照 |

### TC-P002-N002：微信支付完整闭环

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P002-N002 |
| **流程名称** | 微信Native扫码支付正常流程 |
| **前置条件** | 微信支付配置已激活（wxpay profile）；存在可用套餐 |
| **操作步骤** | ① 下单 → ② 发起支付 channel="wxpay" → ③ 返回 codeUrl（二维码内容） → ④ 用户扫码支付 → ⑤ 微信异步回调 → ⑥ 订单履行 → ⑦ 订阅激活 |
| **预期结果** | 回调验签成功 → markSuccess → fulfillOrder → 订阅激活 |
| **注意** | 依赖微信商户号，本地测试可 Mock WeChat API 响应 |

### TC-P002-E001：支付回调重复调用（幂等性）

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P002-E001 |
| **流程名称** | 支付成功回调重复触发 |
| **前置条件** | 支付单已创建，状态 PENDING |
| **操作步骤** | ① POST mockSuccess(paymentNo) → ② 立即再次 POST mockSuccess(paymentNo) |
| **预期结果** | 第一次返回 SUCCESS，订阅激活；第二次直接返回 SUCCESS 不重复履行、不重复创建订阅、不重复扣费 |

### TC-P002-E002：支付超时/取消

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P002-E002 |
| **流程名称** | 创建订单后不支付 |
| **前置条件** | 订单已创建（状态 PENDING） |
| **操作步骤** | ① 创建订单 → ② 不发起支付 → ③ 检查订单状态 |
| **预期结果** | 订单保持 PENDING 状态；已支付的订阅不受影响 |
| **缺陷位置** | 未发现自动取消超时订单的定时任务 |

### TC-P002-E003：履行时订单状态异常

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P002-E003 |
| **流程名称** | 非PAID状态的订单尝试履行 |
| **前置条件** | 订单存在但状态为 PENDING/CANCELLED |
| **操作步骤** | ① POST /api/orders/{orderNo}/fulfill（绕过前端，直接调接口） |
| **预期结果** | 返回警告日志；不履行；不修改订单状态 |

### TC-P002-B001：支付成功但履约失败

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P002-B001 |
| **流程名称** | fulfill接口异常时支付状态与订单状态不一致 |
| **前置条件** | payment-service运行正常；order-service故障/fulfill接口抛异常 |
| **操作步骤** | ① 创建订单 → ② 发起支付 → ③ 在fulfill环节order-service异常 → ④ 检查支付状态和订单状态 |
| **预期结果** | 支付状态=SUCCESS；订单状态=PAID（未变成FULFILLED）；日志记录fulfill异常；**数据不一致：钱已收但订阅未激活** |
| **缺陷等级** | **严重** |
| **缺陷位置** | payment-service.PaymentApplicationService.mockSuccess() — 目前catch后仅日志，缺乏补偿/重试机制 |

### TC-P002-B002：下单时套餐快照锁定

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P002-B002 |
| **流程名称** | 下单后管理员修改套餐价格，已支付订单按快照还是新价格 |
| **前置条件** | 套餐已创建 |
| **操作步骤** | ① 用户下单（金额99）→ ② 管理员修改套餐价格到199 → ③ 用户支付成功 → ④ 检查履行时使用的金额/价格 |

## 流程 P005：RBAC权限管理

### TC-P005-N001：完整RBAC链路

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P005-N001 |
| **流程名称** | 权限→角色→用户→登录→页面鉴权完整链路 |
| **前置条件** | 超级管理员已登录；存在至少一个非超级管理员用户 |
| **操作步骤** | ① 创建权限码 → ② 创建角色并关联权限码 → ③ 创建管理员用户 → ④ 为用户分配角色 → ⑤ 用新用户登录 → ⑥ 检查菜单/权限 |
| **预期结果** | 新用户登录后菜单仅显示有权限的项；无权限页面返回403或隐藏 |

### TC-P005-E001：权限不足访问接口

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P005-E001 |
| **流程名称** | 低权限用户访问高权限接口 |
| **前置条件** | 用户A只有system:user:list权限 |
| **操作步骤** | ① 用户A登录获取Token → ② 尝试访问/admin/system/permissions（需要system:permission:list）→ ③ 尝试DELETE /admin/system/users/{id} |
| **预期结果** | 返回403 Forbidden；日志记录越权尝试 |

### TC-P005-E002：角色被删除后用户权限

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P005-E002 |
| **流程名称** | 删除角色后已分配用户的权限变化 |
| **前置条件** | 用户A有角色R1（含cms:manage权限） |
| **操作步骤** | ① 删除角色R1 → ② 用户A重新登录 → ③ 检查用户权限和菜单 |
| **预期结果** | 用户A的cms:manage权限消失；菜单中CMS管理项隐藏或403 |

### TC-P005-E003：菜单权限绑定测试

| 字段 | 内容 |
|------|------|
| **用例ID** | TC-P005-E003 |
| **流程名称** | 菜单绑定权限后对用户可见性 |
| **前置条件** | 菜单项"订单管理"绑定了order:list权限 |
