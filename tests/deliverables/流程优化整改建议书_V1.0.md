## 致命缺陷优化方案

### FP-001：支付成功→履约失败数据不一致

#### 问题根因

```
链路: payment-service.mockSuccess()
  ① markSuccess()  → 支付状态写入 SUCCESS ✓
  ② fulfillmentClient.fulfillOrder() → Feign调用order-service
  ③ 异常catch → 仅日志输出 ✗ → 支付SUCCESS但订阅未激活
```

**关键问题**：第②步失败后第①步已提交，无回滚机制。

#### 改造方案

**方案A（推荐）：引入可靠事件表 + 定时补偿**

```java
// 1. 新建事件表
CREATE TABLE IF NOT EXISTS payment_compensation_event (
    id              BIGSERIAL PRIMARY KEY,
    payment_no      VARCHAR(64) NOT NULL,
    event_type      VARCHAR(32) NOT NULL,  -- 'FULFILL_ORDER'
    event_status    VARCHAR(16) NOT NULL DEFAULT 'PENDING', -- PENDING/PROCESSED/FAILED
    retry_count     INT NOT NULL DEFAULT 0,
    max_retries     INT NOT NULL DEFAULT 5,
    payload_json    JSONB,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_retry_at   TIMESTAMPTZ
);

// 2. mockSuccess() 改造——先写事件再履约
@Transactional
public CheckoutResponse mockSuccess(String paymentNo) {
    Payment payment = paymentRepository.findByPaymentNo(paymentNo)...;
    
    // 幂等性校验
    if ("SUCCESS".equals(payment.getStatus())) { ... return; }
    
    payment.markSuccess();
    paymentRepository.save(payment);
    
    // 写入补偿事件
    CompensationEvent event = new CompensationEvent(
        paymentNo, "FULFILL_ORDER", payment.getOrderNo());
    compensationEventRepository.save(event);
    
    // 尝试立即执行（同步）
    try {
        fulfillmentClient.fulfillOrder(payment.getOrderNo());
        event.markProcessed();
        compensationEventRepository.save(event);
    } catch (Exception e) {
        log.error("履约失败，补偿事件已记录: paymentNo={}", paymentNo, e);
        // 事件保持PENDING，等待定时任务补偿
    }
    
    return new CheckoutResponse(paymentNo, "SUCCESS", null, null);
}

// 3. 定时补偿任务（@Scheduled）
@Scheduled(fixedRate = 30_000)  // 每30秒扫描
@Transactional
public void compensateFailedEvents() {
    List<CompensationEvent> pendingEvents = 
        compensationEventRepository.findByEventStatusAndRetryCountLessThan(
            "PENDING", 5, PageRequest.of(0, 50));
    
    for (CompensationEvent event : pendingEvents) {
        try {
            var payment = paymentRepository.findByPaymentNo(event.getPaymentNo());
            if (payment == null || !"SUCCESS".equals(payment.getStatus())) {
                event.markFailed();
                compensationEventRepository.save(event);
                continue;
            }
            if ("FULFILL_ORDER".equals(event.getEventType())) {
                fulfillmentClient.fulfillOrder(event.getPayloadJson());
            }
            event.markProcessed();
        } catch (Exception e) {
            event.incrementRetry();
            log.warn("补偿事件重试: paymentNo={}, retry={}/{}", 
                event.getPaymentNo(), event.getRetryCount(), event.getMaxRetries());
        }
        compensationEventRepository.save(event);
    }
}
```

**方案B（简单方案）：异步消息队列**

```java
// 支付成功后发送 MQ 消息
@Transactional
public CheckoutResponse mockSuccess(String paymentNo) {
    // ... 支付状态更新 ...
    
    // 发送消息（事务消息或先提交事务再发MQ）
    eventPublisher.publishEvent(new PaymentSuccessEvent(paymentNo, orderNo));
}

// 异步监听器
@EventListener
@Transactional
public void handlePaymentSuccess(PaymentSuccessEvent event) {
    try {
        fulfillmentClient.fulfillOrder(event.getOrderNo());
    } catch (Exception e) {
        log.error("履约失败，需人工介入: orderNo={}", event.getOrderNo());
        // 发送钉钉/邮件告警
    }
}
```

#### 改造注意事项

| 注意项 | 说明 |
|--------|------|
| 幂等性 | 补偿任务执行前必须校验订单是否已是 FULFILLED 状态 |
| 重试上限 | 达到 max_retries 后标记 FAILED，转人工处理 |
| 告警 | 补偿失败需要实时告警（钉钉/邮件） |
| 上线风险 | 低——仅增加重试机制，不影响现有流程 |
| 回归要点 | 验证正常下单-支付-履约仍正常；验证order-service宕机恢复后补偿执行 |

## 严重缺陷优化方案

### FP-003：PENDING订单无超时取消机制

#### 改造方案

```java
// OrderApplicationService.java — 新增
@Scheduled(fixedRate = 60_000)  // 每分钟扫描
@Transactional
public void autoCancelExpiredOrders() {
    Instant expireThreshold = Instant.now().minus(30, ChronoUnit.MINUTES);
    List<Order> expiredOrders = orderRepository.findByStatusAndCreatedAtBefore(
        OrderStatus.PENDING, expireThreshold, PageRequest.of(0, 100));
    
    for (Order order : expiredOrders) {
        try {
            order.cancel();
            orderRepository.save(order);
            log.info("Auto-cancelled expired order: {}", order.getOrderNo());
        } catch (Exception e) {
            log.error("Failed to auto-cancel order: {}", order.getOrderNo(), e);
        }
    }
}
```

**涉及代码**：
- `OrderRepository` 新增 `findByStatusAndCreatedAtBefore(OrderStatus, Instant, Pageable)`
- `OrderServiceApplication` 添加 `@EnableScheduling`

#### 改造注意事项

| 注意项 | 说明 |
|--------|------|
| 超时时长 | 建议可配置（application.yml: `order.expire-minutes=30`） |
| 分批处理 | 增加 Pageable 控制每次处理量，避免长事务 |
| 幂等保护 | cancel() 已有守卫逻辑（仅PENDING可取消） |

### FP-004：禁用用户后Token未失效

#### 改造方案

```java
// SaasUserApplicationService.java 或 SaasUserController.java

@Transactional
public UserResponse disableUser(Long userId) {
    User user = userRepository.findById(userId)
        .orElseThrow(() -> new IllegalArgumentException("User not found"));
    user.setStatus("disabled");
    userRepository.save(user);
    
    // 强制用户下线 — 使其所有会话立即失效
    StpUtil.logout(userId);
    // 或: StpUtil.kickout(userId); // 踢下线但保留会话历史
    
    log.info("User disabled and logged out: userId={}", userId);
    return toResponse(user);
}
```

#### 改造注意事项

| 注意项 | 说明 |
|--------|------|
| API Key | 禁用用户后API Key也应立即失效（verify-key接口校验status） |
| 踢下线 | 若使用 `kickout` 需合理配置 Sa-Token 的 allow Concurrent 策略 |

### FP-007：删除权限未级联清理角色关联

#### 改造方案

```java
@Transactional
public void deletePermission(Long id) {
    // 先查询所有引用了该权限的角色，解除关联
    List<AdminRole> rolesWithPermission = adminRoleRepository.findAllByPermissions_Id(id);
    for (AdminRole role : rolesWithPermission) {
        role.getPermissions().removeIf(p -> p.getId().equals(id));
        adminRoleRepository.save(role);
    }
    
    // 再删除权限
    permissionRepository.deleteById(id);
}
```

**涉及代码**：
- `AdminRoleRepository` 新增 `findAllByPermissions_Id(Long permissionId)`

### FP-008：管理后台无禁用SaaS用户入口

#### 改造方案

**前端 Admin → Users.vue：**

```vue
<!-- 用户列表追加操作列 -->
<n-button 
  v-permission="'system:user:update'" 
  :type="user.status === 'active' ? 'warning' : 'success'"
  size="small"
  @click="toggleUserStatus(user)">
  {{ user.status === 'active' ? '禁用' : '启用' }}
</n-button>
```

```typescript
async function toggleUserStatus(user: any) {
  const newStatus = user.status === 'active' ? 'disabled' : 'active'
  await request(`/api/users/${user.id}/status?status=${newStatus}`, { method: 'PUT' })
  user.status = newStatus
  window.$message.success(newStatus === 'disabled' ? '已禁用' : '已启用')
}
```

**后端 system-service → AdminUserController：**

```java
@PutMapping("/{id}/status")
@SaCheckPermission("system:user:update")
public ApiResponse<Void> updateUserStatus(@PathVariable Long id, @RequestParam String status) {
    if (!Set.of("active", "disabled").contains(status)) {
        return ApiResponse.error(400, "Invalid status value");
    }
    userServiceClient.updateUserStatus(id, status);
    return ApiResponse.ok();
}
```

### FP-015：管理端审计日志

**使用 Spring AOP + 自定义注解：**

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface AuditLog {
    String action();         // 操作类型
    String resourceType();   // 资源类型
}

@Aspect
@Component
public class AuditLogAspect {
    @Around("@annotation(auditLog)")
    public Object log(ProceedingJoinPoint joinPoint, AuditLog auditLog) throws Throwable {
        // 记录操作前状态
        Object result = joinPoint.proceed();
        // 记录操作后状态 + 差异对比
        // 写入 audit_log 表
        return result;
    }
}
```

使用示例：
```java
@AuditLog(action = "CMS_PUBLISH", resourceType = "CMS_PAGE")
@PostMapping("/{pageId}/translations/{locale}/publish")
public ApiResponse<PageDetailResponse> publish(...) { ... }
```
