## 技能一：业务流程梳理技能

### SK-PTA-001：端到端流程拓扑绘制方法

```
输入: 接口清单 + 功能点清单 + 微服务依赖图
方法:
  ① 识别核心业务闭环（用户→价值→回收）
  ② 沿前端操作→API→微服务→DB链路展开
  ③ 标注前置条件/依赖/后置影响
  ④ 区分主流程/分支流程/边缘流程
输出: 业务流程拓扑图（含状态流转 + 数据流）
```

**复用场景**：任何 SaaS 平台的业务流程测试初始阶段

### SK-PTA-002：流程状态机识别

```
关键检查点:
  □ 状态枚举是否完整（无缺失状态）
  □ 状态转换是否受限（非法转换拦截）
  □ 状态变更是否有时间戳记录
  □ 状态变更是否有审计日志
  □ 并发状态更新是否有锁保护

典型状态机模式:
  订单: PENDING ↔ PAID ↔ FULFILLED ↔ REFUNDED
               ↘ CANCELLED / EXPIRED
  支付: PENDING → PROCESSING → SUCCESS / FAILED / CLOSED
  订阅: active → expired (定时任务)
  用户: enabled ↔ disabled (软删除)
```

### SK-PTA-003：跨服务事务边界识别

```
识别方法:
  ① 一条流程跨越几个 @Transactional 注解？
  ② 每个事务的数据库是什么（同一库还是跨库）？
  ③ 事务之间是否有补偿机制？
  ④ 失败后是回滚、重试、还是标记待补偿？

本项目典型案例:
  下单-支付-履约: 3个@Transactional跨2个服务，无补偿机制 → FP-001
```

## 技能二：流程测试设计技能

### SK-PTD-001：流程测试用例三层设计法

```
第1层 - 正常流程（黄金路径）
  数据: 标准有效数据
  操作: 按产品设计流程严格操作
  预期: 完整闭环、状态符合预期、数据一致

第2层 - 异常流程（断点测试）
  数据: 边界值、空值、非法值
  操作: 流程中断（支付未完成、超时、服务宕机）
  预期: 优雅降级、错误提示明确、不产生脏数据

第3层 - 分支流程（偏离测试）
  数据: 满足分支触发条件的数据
  操作: 撤回、驳回、回滚、取消、重新提交
  预期: 分支状态正确、不影响主线流程
```

### SK-PTD-002：过程依赖矩阵设计

```
测试前必须列出的依赖关系:

+----------------+------------------+------------------+
| 前置操作        | 后置操作          | 依赖验证点       |
+----------------+------------------+------------------+
| 创建套餐组      | 创建套餐          | group_id 外键    |
| 创建套餐        | 创建价格          | plan_id 外键     |
| 创建订单        | 发起支付          | orderNo          |
| 支付成功        | 订阅激活          | 回调幂等性       |
| 角色分配权限    | 用户分配角色      | 权限继承         |
+----------------+------------------+------------------+
```

### SK-PTD-003：数据一致性审计清单

```
流程测试每个环节需审计:
  □ 源数据未变（不变的部分）？
  □ 目标数据正确写入（新增/修改的部分）？
  □ 数据总量正确（无重复创建/无漏删）？
  □ 关联数据同步更新（外键/中间表/快照）？
  □ 状态变更是否在预期范围内？
```

## 技能三：SpringCloud 微服务流程问题优化技能

### SK-PTO-001：Feign调用异常对流程的影响评估

```
Feign调用在不同位置的失败影响:

Feign位置              → 失败影响	             → 建议防护
@Transactional 内部     → 事务回滚，数据一致      → 移出事务外
@Transactional 外部     → 数据已提交，需补偿      → 可靠事件表
回调/异步处理           → 用户无感知             → 告警+重试
同步关键路径            → 用户体验降级            → 熔断+降级+超时配置
```

**配置参考：**

```yaml
# application.yml Feign 客户端配置
spring.cloud.openfeign.client.config:
  default:
    connect-timeout: 5000
    read-timeout: 10000
  order-service:
    connect-timeout: 3000
    read-timeout: 5000
```

### SK-PTO-002：分布式事务补偿模式

```
针对"支付成功但订阅未激活"模式:

┌─────────────────────────────────────────────────────────┐
│ 可靠事件表（Reliable Event Table）模式                    │
│                                                         │
│ 步骤1: 本地事务A 更新支付状态 + 插入补偿事件(PENDING)    │
│ 步骤2: 尝试远程调用B（Feign调用order-service履约）       │
│     成功 → 更新补偿事件为 PROCESSED                     │
│     失败 → 事件保持 PENDING                             │
│ 步骤3: 定时任务扫描 PENDING 事件，重试调用B             │
│ 步骤4: 达到重试上限 → 标记 FAILED → 告警人工处理        │
└─────────────────────────────────────────────────────────┘
```

### SK-PTO-003：幂等性设计模式

```
流程中每个"写"操作都需要幂等性保障:

支付回调:
  if ("SUCCESS".equals(payment.getStatus())) return;  // 已处理，跳过
  
订单履约:
  if (order.getStatus() == OrderStatus.FULFILLED) return;  // 已履行，跳过

余额充值:
  INSERT ... ON CONFLICT DO NOTHING;  // 基于业务订单号唯一约束
  
创建订阅:
  方案A: 先查后建（非原子）
  方案B: INSERT ... ON CONFLICT (user_id, plan_code) DO UPDATE  // 原子
```

### SK-PTO-004：流程状态机防护编码规范

```java
// 正确的状态机模式
public enum OrderStatus {
    PENDING {
        @Override public boolean canTransitionTo(OrderStatus target) {
            return target == PAID || target == CANCELLED || target == EXPIRED;
        }
    },
    PAID {
        @Override public boolean canTransitionTo(OrderStatus target) {
            return target == FULFILLED || target == REFUNDED;
        }
    },
    // ... 每个状态只定义允许的转换
}

// 实体类中的守卫
public void markFulfilled() {
    if (!status.canTransitionTo(OrderStatus.FULFILLED)) {
        throw new IllegalStateException(
            "Cannot fulfill order in status: " + status);
    }
    this.status = OrderStatus.FULFILLED;
    this.fulfilledAt = Instant.now();
}
```

### SK-PTO-005：流程日志埋点规范

```
每个流程关键节点必须输出可追踪日志:

格式规范:
  [流程ID][节点名] 关键数据 耗时

示例:
  [P002][createOrder] orderNo=ORD20260606001 planCode=premium cost=15ms
  [P002][checkout] paymentNo=PAYxxx channel=mock
  [P002][mockSuccess] paymentNo=PAYxxx status=PENDING→SUCCESS
  [P002][fulfillOrder] orderNo=ORD20260606001 status=PAID→FULFILLED

目的:
  - 问题定位时 grep [P002] 即可追踪整条链路
  - 可接入 ELK/Kibana 做流程耗时分析
```
