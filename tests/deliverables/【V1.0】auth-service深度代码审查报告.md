## 二、发现的问题

### CRIT-001 [P1] AdminSystemRoleController 存在重复字段声明

**文件**：`AdminSystemRoleController.java:79`

```java
private final AdminUserRepository adminUserRepository;  // DUPLICATE FIELD
```

该字段在文件中声明了两次（第 49 行和第 79 行），导致编译错误。**这是之前 #5 修复时引入的 bug**。

**影响**：服务无法编译启动。

**修复**：删除重复行。

---

### CRIT-002 [P1] UserController DDD 接口 `getUser` 用 ID 查用户名

**文件**：`UserController.java:50-54`

```java
@GetMapping("/users/{id}")
public ApiResponse<User> getUser(@PathVariable Long id) {
    return authApplicationService.findByUsername(String.valueOf(id))  // ← BUG
```

将 `id` 转成 String 后调用 `findByUsername()`，正确应为 `findById(id)`。

**影响**：DDD 接口 `/api/ddd/auth/users/{id}` 始终返回 404（除非用户名恰好等于数字 ID）。

**修复**：
```java
return authApplicationService.findById(id)
        .map(ApiResponse::ok)
        .orElse(ApiResponse.error(404, "User not found"));
```

---

### CRIT-003 [P1] deletePermission 未级联清理关联

**文件**：`PermissionApplicationService.java:86-88`

```java
@Transactional
public void deletePermission(Long id) {
    permissionRepository.deleteById(id);
}
```

直接删除权限码，未清理 `admin_role_permission` 中间表关联记录。如果该权限已被分配给角色，删除后角色加载权限时可能产生数据不一致。

**修复**：删除前先检查关联并清理。

---

### WARN-001 [P2] 登录失败时错误信息过于具体

**文件**：`AuthApplicationService.java:67`

```java
throw new BadCredentialsException("Invalid username or password");
```

**分析**：错误信息未区分"用户不存在"和"密码错误"，安全性合理（防止枚举）。✅ **最佳实践**。

但 `createPermission` 和 `createUser` 中：

```java
return ApiResponse.error(400, e.getMessage());
```

直接暴露了内部异常信息（如 "Permission code already exists: XXX"），存在信息泄露风险。建议统一使用业务错误码。

---

### WARN-002 [P2] `AdminSystemUserController` 的 `adminUserRepository` 字段标记 `@SuppressWarnings("unused")`

**文件**：`AdminSystemUserController.java:43`

注入但未使用的依赖，建议移除。

---

### WARN-003 [P2] `MenuApplicationService.toResponse()` 时间戳硬编码

```java
OffsetDateTime.now(), OffsetDateTime.now()
```

返回的 `createdAt`/`updatedAt` 字段使用了当前时间而非数据库中存储的实际时间。建议从实体读取。

---

### WARN-004 [P2] `RoleController` 和 `UserController`（DDD 接口）未做任何权限控制

**文件**：`RoleController.java`, `UserController.java`

这些 `/api/ddd/auth/**` 路径通过 Gateway 路由到 auth-service：

```yaml
- id: auth-ddd
  path: /api/ddd/auth/**
  uri: http://127.0.0.1:8082
```

DDD 接口没有任何 `@SaCheckPermission` 或 `StpUtil.checkLogin()` 保护，但 Gateway 和 SaTokenConfig 已放行 `/api/**`。**这些接口虽是内部使用（Feign），但对外暴露存在安全风险。**

---

### WARN-005 [P2] `AdminSystemRoleController.deleteRole` 全表扫描

```java
var usersWithRole = adminUserRepository.findAll();
for (var user : usersWithRole) {
    if (user.getRoles().remove(role)) {
        adminUserRepository.save(user);
    }
}
```

删除角色时遍历所有用户。当用户数量增长时性能急剧下降。应使用 `@Query` 原生 SQL 或 JPA `deleteByRoleId` 方法。

---

### INFO-001 [P3] 日志中打印了密码参数

```java
log.info("Creating user: username={}", username);
```

✅ 做法正确，未打印密码。但调用处：

```java
public record CreateUserRequest(String username, String password, ...)
```

需确认 Spring 的 `@RequestBody` 日志过滤不会泄露密码。

---

### INFO-002 [P3] DTO 使用 Java Record 模式

所有 DTO 均使用 `record` 类型，简洁不可变。✅ **推荐做法**。

### 关键数据流

```
POST /admin/auth/login
  → AdminAuthController.login()
    → AuthApplicationService.login()
      → AuthUserService.loadUserByUsername()  [加载 AdminUser + roles + permissions]
      → passwordEncoder.matches()              [验证密码]
      → StpUtil.login()                        [创建会话]
      → SaSession.set(PERMISSION_LIST, ...)    [缓存权限]
      → LoginResponse(token, user, permissions)
```
