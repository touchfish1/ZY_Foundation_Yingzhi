## 优先级排序标准

| 优先级 | 处理时限 | 说明 |
|--------|---------|------|
| **P0-紧急** | 立即修复 | 安全漏洞/核心流程阻断 |

## P0-阻塞：必须立即修复

### ISSUE-001：Mock 支付接口安全风险

| 维度 | 内容 |
|------|------|
| **问题描述** | `POST /api/payments/mock/{paymentNo}/success` 接口无任何鉴权保护，知道 paymentNo 即可模拟支付成功。生产环境未禁用时可直接绕过支付。 |
| **风险等级** | **P0-阻塞** |
| **影响范围** | 支付模块、订单模块、订阅模块 |
| **问题原因** | 开发/测试用 Mock 接口未做环境隔离和权限控制 |
| **优化方案** | **方案A（推荐）**：通过 Spring Profile 条件加载，仅开发/测试 Profile 暴露此接口：`@Profile("dev", "test")` |
| | **方案B**：保留接口但增加鉴权（需 admin token）+ IP 白名单 |
| | **方案C**：Gateway 层过滤，生产环境路由不转发该路径 |
| **代码修改建议** | `PaymentPublicController.java` 增加 `@Profile({"dev", "test"})` 条件注解；或在 Gateway 路由配置排除该路径 |
| **上线风险** | 无 — 仅增强安全性，不影响正常业务流程 |

## P1-严重：本迭代内修复

### ISSUE-002：订单支付回调非幂等

| 维度 | 内容 |
|------|------|
| **问题描述** | 支付成功回调(P02 → fulfill)未做幂等性防护。重复调用会导致：① 重复创建订阅；② 多次扣减余额；③ 重复发货 |
| **风险等级** | **P1-严重** |
| **影响范围** | order-service, payment-service |
| **问题原因** | 回调处理逻辑未检查 payment_transaction.status 状态机 |
| **优化方案** | 1. payment_transaction 增加状态机校验：已 SUCCESS 状态的直接返回成功不处理 |
| | 2. fulfill 接口使用 `@Transactional` + 唯一约束防止重复 |
| | 3. 关键操作添加分布式锁(Redisson) |
| **代码修改建议** | `PaymentPublicController.mockSuccess()` 中增加：`if (!"PENDING".equals(tx.getStatus())) return success;` |

---

### ISSUE-003：余额并发扣款无乐观锁

| 维度 | 内容 |
|------|------|
| **问题描述** | saas_user 表无 @Version 字段，高并发下扣款可能发生超扣（余额扣为负数） |
| **风险等级** | **P1-严重** |
| **影响范围** | user-service, 所有消费型接口 |
| **问题原因** | JPA 缺少乐观锁，扣款非原子操作 |
| **优化方案** | 1. saas_user 表增加 `@Version private Long version;` 字段 |
| | 2. 扣款改为原子 SQL：`UPDATE saas_user SET balance = balance - ? WHERE id = ? AND balance >= ?` |
| | 3. 增加补偿机制：若扣款后余额 < 0 触发回滚 |
| **代码修改建议** | SaasUser entity 增加 `@Version` 字段；BalanceService 扣款方法使用 `@Lock(PESSIMISTIC_WRITE)` 或原生 SQL |

---

### ISSUE-004：管理员删除用户未校验业务关联

| 维度 | 内容 |
|------|------|
| **问题描述** | 删除 admin_user 时未检查 `cms_page.created_by`、`asset_file.created_by`、`cms_publish_record.operator_id` 等外键引用，导致孤立记录 |
| **风险等级** | **P1-严重** |
| **影响范围** | auth-service CMS模块 |
| **问题原因** | 物理删除前缺少关联检查 |
| **优化方案** | **方案A**：改为软删除（`status=disabled`），保留用户基础信息，前端显示"已注销" |
| | **方案B**：删除前检查所有关联表，有引用时拒绝删除并返回关联明细 |
| **代码修改建议** | AdminSystemUserController.deleteUser() 增加逻辑：① 查询 cms_page/cms_publish_record/asset_file 是否有关联记录；② 有则抛出 BusinessException 提示"该用户存在关联数据，无法删除" |

---

### ISSUE-005：角色删除后权限残留

| 维度 | 内容 |
|------|------|
| **问题描述** | 删除角色后 admin_user_role 表中用户-角色关联记录未清理，导致已分配角色的用户加载角色信息时可能 NPE |
| **风险等级** | **P1-严重** |
| **影响范围** | auth-service |
| **问题原因** | 角色删除未级联清理关联表 |
| **优化方案** | 在 `@Transactional` 删除角色时，先清理 admin_user_role 和 admin_role_permission |
| **代码修改建议** | `RoleServiceImpl.deleteRole(id)` 增加：`adminUserRoleRepository.deleteByRoleId(id); adminRolePermissionRepository.deleteByRoleId(id);` |

---

### ISSUE-006：API Key 安全性不足

| 维度 | 内容 |
|------|------|
| **问题描述** | ① API Key 生成算法可能可预测；② verify-key 接口无速率限制可暴力枚举；③ 旧 Key 重新生成后旧 Key 可能未立即失效 |
| **风险等级** | **P1-严重** |
| **影响范围** | user-service |
| **优化方案** | 1. 使用 `SecureRandom` + `Base64URL` 生成 32 字节以上随机 Key |
| | 2. verify-key 接口增加 Guava RateLimiter（每分钟 100 次/每 IP） |
| | 3. 重新生成 Key 时立即标记旧 Key 失效并清理缓存 |
| **代码修改建议** | `ApiKeyService.java`：`SecureRandom.getInstanceStrong()`；`VerifyKeyController` 增加 `@RateLimiter(name = "verifyKey")` |

---

### ISSUE-007：双服务数据库重复定义

| 维度 | 内容 |
|------|------|
| **问题描述** | `order_main` 和 `payment_transaction` 表在两个服务（api BFF + order-service/payment-service）中都有 Flyway migration 定义。微服务拆分过渡阶段若不处理，独立部署时将出现数据分裂 |
| **风险等级** | **P1-严重** |
| **影响范围** | 架构层面，order-service, payment-service, api |
| **优化方案** | **短期过渡**：api 模块的 migration 中移除重复表定义，改为只通过 Feign 调用 order-service/payment-service |
| | **长期方案**：每个表一个微服务独享，其他服务通过 RPC 访问 |
| **代码修改建议** | `api/src/main/resources/db/migration/` 移除 V004 中 order_main / payment_transaction 的表定义；改 Feign 调用 order-service |

---

### ISSUE-008：前端 Token 体系未区分

| 维度 | 内容 |
|------|------|
| **问题描述** | 对外网站和管理后台使用两套用户体系(SaaS User / Admin User)，但 Token 存储和传递未做区分，可能混用 |
| **风险等级** | **P1-严重** |
| **影响范围** | 两个前端项目 + Gateway |
| **优化方案** | 1. 存储 key 分离：前端 `localStorage` 使用 `saas_token` / `admin_token` |
| | 2. API 层根据 `api/` 和 `admin/` 前缀自动选择 Token |
| | 3. Gateway 层校验 Token 时根据路径解析 type 属性 |
| **代码修改建议** | `frontend/web/composables/useSaasAuth.ts` 存储使用 `saas_token`；`frontend/admin/src/api/http.ts` 使用 `admin_token` |

| 维度 | 内容 |

### ISSUE-010：CMS 页面 slug 唯一性校验

| 维度 | 内容 |
|------|------|
| **问题描述** | 创建页面时 slug 重复直接抛数据库异常，返回 500 错误 |
| **影响范围** | api (CMS模块) |
| **优化方案** | 应用层预校验 `cmsPageRepository.findBySlug(slug)` 是否存在，返回 409 "slug 已被使用" |

---

### ISSUE-011：发布草稿未校验内容完整性

| 维度 | 内容 |
|------|------|
| **问题描述** | 直接发布空草稿或含无效区块类型的页面可能引发 NPE |
| **影响范围** | api (CMS模块) |
| **优化方案** | 发布前校验：`draft_version_id` 不为空；`content_json` 中 block type 均为有效注册类型 |
| **代码位置** | `CmsPageTranslationService.publish()` |

---

### ISSUE-012：多语言 locale 未做白名单

| 维度 | 内容 |
|------|------|
| **问题描述** | locale 参数未限制取值范围，可写入任意字符串值 |
| **影响范围** | api (CMS模块) |
| **优化方案** | 使用 `@Pattern(regexp = "^[a-z]{2}-[A-Z]{2}$")` 或枚举白名单 `zh-CN/en-US/ja-JP` |
| **代码位置** | `CmsPageTranslation.java` 的 locale 字段 |

---

### ISSUE-013：价格金额未做业务校验

| 维度 | 内容 |
|------|------|
| **问题描述** | `POST /admin/product/prices` 的 amount 缺少 `@Positive` 和精度校验 |
| **影响范围** | api (产品模块) |
| **优化方案** | 增加 `@DecimalMin("0.01") @DecimalMax("9999999.99") @Digits(integer=7, fraction=2)` |

---

### ISSUE-014：订单快照数据一致性

| 维度 | 内容 |
|------|------|
| **问题描述** | 订单履行时使用 snapshot_json 还是最新套餐配置？当前逻辑不明确 |
| **影响范围** | order-service |
| **优化方案** | 明确策略并加注释：履行时严格使用 `snapshot_json`，新增 "定价锁定于下单时" 文档说明 |

---

### ISSUE-015：重复注册邮箱提示不友好

| 维度 | 内容 |
|------|------|
| **问题描述** | 同邮箱重复注册时返回 500 而非友好的 409 |
| **影响范围** | user-service |
| **优化方案** | 注册前查询 email 是否存在：`saasUserRepository.findByEmail(email).ifPresent(() -> throw new BusinessException(409, "该邮箱已被注册"))` |

---

### ISSUE-016：系统设置值未限制长度

| 维度 | 内容 |
|------|------|
| **问题描述** | setting_value 为 text 类型，无长度限制，可写入超长数据 |
| **影响范围** | system-service |
| **优化方案** | 对 text 字段增加 `@Size(max = 65535)` 校验；对短文本设置单独限制 |

---

### ISSUE-017：文件上传缺少校验

| 维度 | 内容 |
|------|------|
| **问题描述** | 文件上传未限制 MIME 类型和大小 |
| **影响范围** | api (资产模块) |
| **优化方案** | 1. 白名单 MIME 类型：image/*, application/pdf, text/* |
| | 2. 限制大小：`spring.servlet.multipart.max-file-size=50MB` |
| | 3. UUID 重命名文件防止路径覆盖 |
| **代码位置** | `AdminAssetController.uploadFile()` |

---

### ISSUE-018：统一异常处理

| 维度 | 内容 |
|------|------|
| **问题描述** | 各微服务未统一异常返回格式，DB 异常可能暴露堆栈 |
| **影响范围** | 全部微服务 |
| **优化方案** | 每个微服务实现 `@RestControllerAdvice` 全局异常处理器，定义 `ErrorResponse` 统一格式 |

---

### ISSUE-019：管理端操作审计日志缺失

| 维度 | 内容 |
|------|------|
| **问题描述** | CMS发布、产品修改、权限变更等管理操作未记录审计日志 |
| **影响范围** | api CMS模块, auth-service, 产品模块 |
| **优化方案** | 使用 AOP 或 Spring 事件机制对管理端所有增加/修改/删除操作记录审计日志 |
